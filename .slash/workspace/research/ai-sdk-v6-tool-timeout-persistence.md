# Research: Vercel AI SDK v6 - Tool Execution Patterns

**Date**: 2026-04-13

## Summary

This research covers three advanced patterns in Vercel AI SDK v6: handling long-running tool execution with timeouts and progress updates, proper serialization/persistence of messages with tool invocations, and parallel tool execution capabilities.

---

## 1. Long-Running Tool Execution

### Q: What happens when a tool takes 60+ seconds? Does the stream timeout?

**Answer**: Yes, AI SDK v6/v7 has built-in timeout support:

```typescript
// Set default timeout for all tools (5 seconds)
const result = await generateText({
  model: openai('gpt-4o'),
  tools: { slowTool, fastTool },
  timeout: {
    toolMs: 5000,  // 5 second default for all tools
  },
});

// Per-tool overrides
const result = await generateText({
  model: openai('gpt-4o'),
  tools: { slowTool, fastTool },
  timeout: {
    toolMs: 5000,           // default
    tools: {
      slowToolMs: 30000,   // 30 seconds for slowTool
      fastToolMs: 2000,    // 2 seconds for fastTool
    },
  },
});
```

**Evidence** ([AI SDK Docs](https://github.com/vercel/ai/blob/main/content/docs/03-ai-sdk-core/25-settings.mdx)):
> Use toolMs to set a default timeout for all tool executions. Tools exceeding this timeout abort and return a tool-error.

### Q: Can we return partial results?

**Answer**: Yes, use **AsyncIterable** in the tool's `execute` function:

```typescript
import { tool } from 'ai';
import { z } from 'zod';

const longRunningTool = tool({
  description: 'Process large dataset',
  inputSchema: z.object({
    datasetId: z.string(),
  }),
  async *execute({ datasetId }) {
    // Yield intermediate progress
    yield {
      status: 'loading' as const,
      text: 'Connecting to dataset...',
    };

    await processStep1(datasetId);

    yield {
      status: 'loading' as const,
      text: 'Processing records...',
      progress: 50,
    };

    await processStep2(datasetId);

    // Final result
    yield {
      status: 'success' as const,
      text: 'Processing complete!',
      result: { records: 1000, duration: '45s' },
    };
  },
});
```

**Evidence** ([AI SDK Docs](https://github.com/vercel/ai/blob/main/content/docs/03-ai-sdk-core/15-tools-and-tool-calling.mdx)):
> Return an AsyncIterable from the execute function to stream status updates or intermediate data. The final value yielded by the generator is treated as the ultimate tool result.

### Q: Can we use `toolCallStreaming` for progress updates?

**Answer**: The `experimental_toolCallStreaming` option streams tool call **input** (arguments) as they're being generated by the model, not progress during execution. For progress updates, use the AsyncIterable pattern above.

### Q: Fire-and-forget pattern?

**Answer**: Not built-in. Tools are always awaited. For background jobs:

```typescript
// Manual approach - trigger background work, return immediately
const backgroundTool = tool({
  async execute({ taskId }) {
    // Start background job, don't await
    backgroundJobProcessor.start(taskId);

    // Return job ID for polling
    return { jobId: taskId, status: 'started' };
  },
});

// Then use a separate polling tool
const checkJobTool = tool({
  async execute({ jobId }) {
    const status = await backgroundJobProcessor.check(jobId);
    return status;
  },
});
```

**Recommendation**: For 60+ second operations, consider:
1. **Use AsyncIterable** for progress updates (if job can yield progress)
2. **Use timeout config** to set appropriate limits (v7 supports up to custom values)
3. **Use background jobs + polling** for truly long operations (minutes/hours)
4. **Use `experimental_onToolCallStart/Finish`** for monitoring

## 2. Full Message Persistence with Parts

### Q: How to serialize messages with tool invocations to database?

**Answer**: In AI SDK v6, messages use `parts[]` array instead of `content`:

```typescript
import { UIMessage, ModelMessage } from 'ai';

// UIMessage structure (for UI display)
interface UIMessage {
  id: string;
  role: 'system' | 'user' | 'assistant';
  parts: UIMessagePart[];
}

// Part types
type UIMessagePart =
  | { type: 'text'; text: string; state?: 'streaming' | 'done' }
  | { type: 'reasoning'; text: string; state?: 'streaming' | 'done' }
  | { type: 'tool-invocation'; toolInvocation: ToolInvocation }
  | { type: 'file'; mediaType: string; data: string; filename?: string }
  | { type: 'source'; source: { url: string; title?: string } }
  | { type: 'data-custom'; data: unknown };

// Tool invocation states
type ToolInvocationState =
  | 'input-streaming'   // Tool called, args being generated
  | 'input-available'   // Args complete, not yet executed
  | 'output-available'  // Execution complete
  | 'output-error';     // Execution failed
```

**Evidence** ([AI SDK Migration Guide](https://github.com/vercel/ai/blob/main/content/docs/08-migration-guides/26-migration-guide-5-0.mdx)):
> UIMessages (formerly Message) now use a parts array instead of a content property to support complex message structures.

### Recommended Schema (from official example)

The official [ai-sdk-persistence-db](https://github.com/vercel-labs/ai-sdk-persistence-db) example uses this schema:

```typescript
// Three tables: chats, messages, parts

// Chats table
const chats = pgTable('chats', {
  id: varchar().primaryKey(),
});

// Messages table
const messages = pgTable('messages', {
  id: varchar().primaryKey(),
  chatId: varchar().references(() => chats.id),
  role: varchar().$type<'user' | 'assistant'>(),
  createdAt: timestamp(),
});

// Parts table (polymorphic - prefix-based columns)
const parts = pgTable('parts', {
  id: varchar().primaryKey(),
  messageId: varchar().references(() => messages.id),
  type: varchar(),  // 'text', 'reasoning', 'tool-invocation', 'file', etc.
  order: integer(),

  // Text parts
  text_text: text(),

  // Reasoning parts
  reasoning_text: text(),

  // Tool parts (stores tool call ID and state)
  tool_toolCallId: varchar(),
  tool_state: varchar().$type<ToolInvocationState>(),
  tool_errorText: varchar(),
  // Tool-specific input/output columns (e.g., tool_weather_input, tool_weather_output)

  // File parts
  file_mediaType: varchar(),
  file_filename: varchar(),
  file_url: varchar(),
});
```

**Evidence** ([vercel-labs/ai-sdk-persistence-db](https://github.com/vercel-labs/ai-sdk-persistence-db)):
> The schema employs a prefix convention for message part columns to categorize and store different types of content within a message. `tool_[toolName]_*` columns are specifically for tool calls.

### Serialization Helper

```typescript
function serializeMessageParts(parts: UIMessagePart[]): DBPart[] {
  return parts.map((part, index) => {
    switch (part.type) {
      case 'text':
        return {
          type: 'text',
          order: index,
          text_text: part.text,
        };
      case 'tool-invocation':
        return {
          type: 'tool-invocation',
          order: index,
          tool_toolCallId: part.toolInvocation.toolCallId,
          tool_state: part.toolInvocation.state,
          // Store input/output based on state
          ...(part.toolInvocation.state === 'output-available' && {
            [`tool_${part.toolInvocation.toolName}_input`]: JSON.stringify(part.toolInvocation.input),
            [`tool_${part.toolInvocation.toolName}_output`]: JSON.stringify(part.toolInvocation.result),
          }),
        };
      // ... handle other types
    }
  });
}
```

### Restoring to UIMessage

```typescript
function restoreFromDB(messages: DBMessage[], parts: DBPart[]): UIMessage[] {
  return messages.map(msg => ({
    id: msg.id,
    role: msg.role,
    parts: parts
      .filter(p => p.messageId === msg.id)
      .sort((a, b) => a.order - b.order)
      .map(part => {
        switch (part.type) {
          case 'text':
            return { type: 'text' as const, text: part.text_text };
          case 'tool-invocation':
            return {
              type: 'tool-invocation' as const,
              toolInvocation: {
                toolCallId: part.tool_toolCallId,
                toolName: /* extract from column name */,
                state: part.tool_state,
                input: /* parse from column */,
                result: /* parse from column */,
              },
            };
          // ...
        }
      }),
  }));
}
```

### Key Points

- **No built-in serialization** - implement manually or use the official example as reference
- **Store tool state** - `input-streaming`, `input-available`, `output-available`, `output-error`
- **Use prefix columns** - `tool_{toolName}_input`, `tool_{toolName}_output` for type-safe storage
- **Convert for API** - use `convertToModelMessages()` when passing to `streamText`

## 3. Parallel Tool Execution

### Q: Can `streamText` execute multiple tools in parallel?

**Answer**: Yes, but it's **model-dependent**. The LLM decides whether to call tools in parallel:

```typescript
const result = await generateText({
  model: openai('gpt-4o'),
  tools: { getWeather, getStock },
  prompt: 'What is the weather in Paris and Apple stock price?',
});

// Model may call both tools in parallel if supported
console.log(result.toolCalls);
// Potentially: [
//   { toolName: 'getWeather', input: { city: 'Paris' } },
//   { toolName: 'getStock', input: { symbol: 'AAPL' } }
// ]
```

**Evidence** ([AI SDK Cookbook](https://github.com/vercel/ai/blob/main/content/docs/05-node/51-call-tools-in-parallel.mdx)):
> Some language models support calling tools in parallel. This is particularly useful when multiple tools are independent of each other and can be executed in parallel during the same generation step.

### Manual Parallel Execution

For full control, execute manually:

```typescript
// Inside manual agent loop
const toolPromises = toolCalls.map(async (toolCall) => {
  const result = await executeTool(toolCall);
  return {
    role: 'tool' as const,
    content: [{
      toolName: toolCall.toolName,
      toolCallId: toolCall.toolCallId,
      type: 'tool-result' as const,
      output: { type: 'text' as const, value: result },
    }],
  };
});

const toolResults = await Promise.all(toolPromises);
```

**Evidence** ([AI SDK Cookbook](https://github.com/vercel/ai/blob/main/content/docs/05-node/55-manual-agent-loop.mdx)):
> Parallel tool execution allows you to run multiple tools concurrently for better performance. By mapping over tool calls and executing them as promises, you can collect results using Promise.all().

### Recommendation

- **Let the model decide** - Most modern models (GPT-4, Claude 3.5+) support parallel calls
- **Design tools to be independent** - Parallel execution works best when tools don't depend on each other's output
- **Manual parallel** - Use `Promise.all` for explicit control or when model doesn't parallelize well
